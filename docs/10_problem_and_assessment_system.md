# Problem and Assessment System

## Purpose of This Document
This document defines how problems, practice, reasoning tasks, and unlock assessments should work across Math Journey.

The goal is to make assessment feel coherent, rigorous, and scalable.

## Core Principle
Problems are not filler. They are one of the main engines of learning.

## Problem Categories

### Warm-Up
Purpose:
- restore prerequisites
- reduce fear at the start of practice

Typical traits:
- short
- direct
- low cognitive overhead

### Core Practice
Purpose:
- reinforce the main lesson objective

Typical traits:
- standard difficulty
- aligned with worked examples
- enough repetition for fluency

### Reasoning
Purpose:
- test understanding beyond imitation

Typical traits:
- explain
- compare
- justify
- diagnose an error
- complete a proof skeleton

### Application
Purpose:
- connect the concept to a real or disciplinary context

Typical traits:
- word problem
- diagram interpretation
- physics or measurement context
- historical use case

### Challenge
Purpose:
- stretch advanced learners without blocking core progression

Typical traits:
- less scaffolded
- more synthesis
- optional in early versions unless marked otherwise

## Difficulty Labels
The system should use a small, stable set of difficulty labels:
- `warm-up`
- `core`
- `stretch`

This is preferable to a large or game-like ranking system.

## Practice Set Composition
A standard chapter practice set should usually include:
- 1 to 2 warm-up problems
- 3 to 5 core problems
- 1 reasoning prompt
- 1 application prompt when appropriate
- 0 to 2 stretch problems

## Worked Example Relationship
Problems should connect directly to the lesson.

Avoid:
- assigning practice that introduces a new problem type before it has been modeled
- repeating only trivial clones of the worked examples

Prefer:
- recognizable structure with increasing independence
- one or two transfer steps beyond the examples

## Hint System
Hints should reveal help in layers:
1. conceptual nudge
2. setup hint
3. first computational step
4. full solution

Do not reveal the entire answer immediately unless the user chooses that path deliberately.

## Feedback Principles
Feedback should identify the type of issue where possible:
- arithmetic slip
- notation confusion
- setup error
- conceptual misunderstanding
- incomplete reasoning

The tone of feedback should be direct and constructive.

## Unlock Quiz Rules
Every core chapter should include a short unlock quiz.

Recommended structure:
- 4 to 8 questions
- mostly core objective coverage
- at least one conceptual or interpretation question
- minimal trivia

Recommended threshold:
- 80 percent minimum score

## What Unlock Quizzes Should Test
Unlock quizzes should verify:
- the learner can identify the core idea
- the learner can perform the basic method
- the learner can avoid the chapter’s main misconception

They should not try to replace the practice set.

## Reasoning Task Rules
Reasoning tasks may be:
- short written explanation
- choose and justify
- compare two solutions
- identify the mistake in a false solution
- complete a partial argument

They should be meaningful, but not so open-ended that the learner cannot tell what counts as success.

## Problem Authoring Guidance
Each problem should have:
- a clear objective
- a difficulty label
- an answer or expected response
- a hint path if the format supports it

## First-Version Assessment Scope
For the first version, prioritize:
- static worked examples
- static practice sets
- a simple unlock quiz
- one reasoning task

Do not overbuild analytics or adaptive scoring before the chapter model works.

## Definition of Done for a Practice Section
A practice section is ready when:
- it matches the chapter objectives
- it covers straightforward and slightly transferred use
- it includes at least one reasoning element
- it has enough problems to establish competence

## Final Rule
Assessment should feel earned, fair, and clearly tied to what was taught.
